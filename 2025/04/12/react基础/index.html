组件化

整个逻辑其实可以看做一个整体，那么我们就可以将其封装成一个组件：

  • root.render 参数是一个 HTML 元素或者一个组件；
  • 所以我们可以先将业务逻辑封装到一个组件中，然后传入到 ReactDOM.render 函数中的第一个参数

在 React 中，如何封装一个组件呢？有类组件和函数式组件两种 :

类组件主要使用

  • 定义一个类（类名大写，组件的名称是必须大写的，小写会被认为是 HTML 元素），继承自 React.Component
  • 实现当前组件的 render 函数
    • render 当中返回的 jsx 内容，就是之后 React 会帮助我们渲染的内容

数据依赖

在组件中的数据，我们可以分成两类：

  • 参与界面更新的数据：当数据变量时，需要更新组件渲染的内容；
  • 不参与界面更新的数据：当数据变量时，不需要更新将组建渲染的内容；

参与界面更新的数据我们也可以称之为是参与数据流，这个数据是定义在当前对象的 state中

  • 我们可以通过在构造函数中 this.state = {}
  • 当我们的数据发生变化时，我们可以调用 this.setState 来更新数据，并且通知 React 进行 update 操作；
    • 在进行 update 操作时，就会重新调用 render 函数，并且使用最新的数据，来渲染界面

事件绑定

在类中直接定义一个函数，并且将这个函数绑定到元素的 onClick 事件上，当前这个函数的 this 指向的是谁呢？

  • 默认情况下是 undefined
  • 因为在正常的 DOM 操作中，监听点击，监听函数中的 this 其实是节点对象（比如说是 button 对象）
  • 这次因为 React 并不是直接渲染成真实的 DOM，我们所编写的 button 只是一个语法糖，它的本质 React 的 Element 对象

那么在这里发生监听的时候，react 在执行函数时并没有绑定 this，默认情况下就是一个 undefined

  • 我们在绑定的函数中，可能想要使用当前对象，比如执行 this.setState 函数，就必须拿到当前对象的 this
  • 我们就需要在传入函数时，给这个函数直接绑定 this onClick={this.changeText.bind(this)}

JSX语法

认识jsx

JSX 是 React 的核心语法，它允许你在 JavaScript 中编写类似 HTML 的结构。

JSX（JavaScript XML）是 JavaScript 的语法扩展，不是 HTML，也不是模板语言。它的核心作用：

  1. 声明式描述 UI：直观表达组件的结构和数据。
  2. 编译为 JavaScript：JSX 会被转译为 **React.createElement()** 调用，生成 React 元素（即虚拟 DOM 对象）。

为什么 React 使用 JSX

React 认为渲染逻辑本质上与其他 UI 逻辑存在内在耦合

  • 比如 UI 需要绑定事件（button、a 原生等等）；
  • 比如 UI 中需要展示数据状态；
  • 比如在某些状态发生改变时，又需要改变 UI；

他们之间是密不可分，所以 React 没有将标记分离到不同的文件中，而是将它们组合到了一起，这个地方就是组件

**JSX 的书写规范： **

JSX 的顶层只能有一个根元素，所以我们很多时候会在外层包裹一个 div 元素（或者使用后面我们学习的 Fragment）；

  • 为了方便阅读，我们通常在 jsx 的外层包裹一个小括号()，这样可以方便阅读，并且 jsx 可以进行换行书写；
  • JSX 中的标签可以是单标签，也可以是双标签；

注意：如果是单标签，必须以/>结尾；

jsx的使用

1. 标签与元素

  • 普通 HTML 标签：直接使用小写标签名（如 **<div>**、**<span>**）。
  • React 组件：使用大写字母开头的自定义标签（如 **<MyComponent />**）。

2. 嵌入 JavaScript 表达式

  • 使用 **{}** 包裹任何 JavaScript 表达式（变量、函数调用、三元运算符等）。

3. 属性（Props）

  • HTML 属性名：使用驼峰命名（如 **className** 代替 **class**，**htmlFor** 代替 **for**）。
  • 动态属性值：用 **{}** 包裹表达式。
  • 布尔属性：省略值时默认为 **true**（如 **<input disabled />** 等价于 **disabled={true}**）。

4. 子元素

  • 单标签：必须闭合（如 **<img />**）。
  • 多行嵌套：使用 **()** 包裹，避免自动插入分号。

5. 注释

  • 使用 **{/* ... */}** 包裹注释。

  • 方式一：获取 DOM 原生，添加监听事件；

在 React 中是如何操作呢？我们来实现一下 React 中的事件监听，这里主要有两点不同

  • 原因是 btnClick 函数并不是我们主动调用的，而且当 button 触发时，React 内部调用了 btnClick 函数； 而它内部调用时，并不知道要如何绑定正确的 this；

如何解决 this 的问题呢？

  • 方案一：bind 给 btnClick 显示绑定 this
  • 方案二：使用 ES6 class fields 语法
  • 方案三：事件监听时传入箭头函数（推荐）

event 和参数传递

在执行事件函数时，有可能我们需要获取一些参数信息：比如 event 对象、其他参数

  • 情况一：获取 event 对象
    • 很多时候我们需要拿到 event 对象来做一些事情（比如阻止默认行为）
    • 那么默认情况下，event 对象有被直接传入，函数就可以获取到 event 对象；
  • 情况二：更多参数
    • 有更多参数时，我们最好的方式就是传入一个箭头函数，主动执行的事件函数，并且传入相关的其他参数；

jsx本质

实际上，jsx 仅仅只是** React.createElement(component, props, …children) 函数的语法糖**。 所有的 jsx 最终都会被转换成 React.createElement 的函数调用。

createElement 需要传递三个参数：

  • **参数一：type **
    • 当前 ReactElement 的类型； 如果是标签元素，那么就使用字符串表示 “div”； 如果是组件元素，那么就直接使用组件的名称；
  • **参数二：config **
    • 所有 jsx 中的属性都在 config 中以对象的属性和值的形式存储； 比如传入 className 作为元素的 class；
  • **参数三：children **
    • 存放在标签中的内容，以 children 数组的方式进行存储；

react 的虚拟 DOM

1.初次渲染

当你使用 JSX 或 React.createElement 创建 React 元素时(这些元素就是 VNode)，React 会依据这些React.createElement 对象组成一个 JavaScript 对象树 , 这个JavaScript 的对象树就是虚拟 DOM 树（Virtual DOM）

  • 将 jsx 结构或者 React.createElement 创建的对象进行打印；即可查看虚拟 dom 树结构
  • 我们所书写的 Jsx 结构最终会被转换一个嵌套的 ReactElement 对象(虚拟 Dom 树结构) , 最后在页面渲染成真实 Dom 结构

2. 数据更新

当组件的状态（state）或者属性（props）发生改变时，React 会重新构建一个新的虚拟 DOM 树。

3. 差异比较（Diffing）

React 会将新的虚拟 DOM 树和旧的虚拟 DOM 树进行对比，找出两者之间的差异。这个过程被称作 Diffing 算法，其目的是尽可能高效地找出需要更新的部分。

4. 批量更新真实 DOM

React 依据差异比较的结果，只对真实 DOM 中需要更新的部分进行批量更新。这样可以减少直接操作真实 DOM 的次数，从而提升性能。

tip: 虚拟 dom 的主要作用

  • 避免数据发生变化时去刷新整个页面,而是使用 diff 寻找差异,仅部分更新
  • 使用虚拟 dom 过渡,可以跨平台进行不同的渲染

**虚拟 DOM 帮助我们从命令式编程转到了声明式编程的模式 **

React 官方的说法：Virtual DOM 是一种编程理念。

  • 在这个理念中，UI 以一种理想化或者说虚拟化的方式保存在内存中，并且它是一个相对简单的 JavaScript 对象
  • 我们可以通过 root.render 让 虚拟 DOM 和 真实 DOM 同步起来，这个过程中叫做协调（Reconciliation）；

这种编程的方式赋予了 React 声明式的 API：

  • 你只需要告诉 React 希望让 UI 是什么状态；
  • React 来确保 DOM 和这些状态是匹配的；
  • 你不需要直接进行 DOM 操作，就可以从手动更改 DOM、属性操作、事件处理中解放出来；

React组件化开发

组件类型划分

React 的组件相对于 Vue 更加的灵活和多样，按照不同的方式可以分成很多类组件：

  • 根据组件的定义方式，可以分为：**函数组件(Functional Component )和类组件(Class Component)**；
  • 根据组件内部是否有状态需要维护，可以分成：无状态组件(Stateless Component )和有状态组件(Stateful Component)；
  • 根据组件的不同职责，可以分成：展示型组件(Presentational Component)和容器型组件(Container Component)；

这些概念有很多重叠，但是他们最主要是关注数据逻辑和 UI 展示的分离：

  • 函数组件、无状态组件、展示型组件主要关注 UI 的展示；
  • 类组件、有状态组件、容器型组件主要关注数据逻辑；

当然还有很多组件的其他概念：比如异步组件、高阶组件等

  • 使用 **this.state** 初始化状态，通过 **this.setState()** 更新。

函数式组件

React 团队明确表示不会废弃类组件，但新特性（如并发模式）优先支持函数组件 ,函数式组件 + Hooks 几乎可以替代类组件 , 但极少数场景（如错误边界）仍需类组件

生命周期

React 类组件的生命周期可分为三大阶段：

  1. 挂载阶段（Mounting）：组件被创建并插入 DOM
  2. 更新阶段（Updating）：组件因 props 或 state 变化重新渲染
  3. 卸载阶段（Unmounting）：组件从 DOM 中移除

setState

在 vue 中修改响应式数据 , 会触发数据劫持 , 将 templete 模板 转成 h 函数进行重新渲染 ,而在 react 中并不能直接通过修改 state 来触发 数据的重新渲染, 必须通过 setState 修改来告诉 react 数据发生变化, 当触发 setState 时会重新调用 render 函数渲染新的数据

基本使用

  • 在 setState 中并不是直接替换掉原来的数据(否则会丢失数据,如 message),而是使用了 Object.assign 进行了合并操作

**setState**** 的两种形式**

** 对象形式**

这种形式适用于当你要更新的状态不依赖于当前状态时。

** 函数形式**

  • 当更新状态依赖于当前状态时，建议使用函数形式。因为 React 可能会将多个 setState 调用合并，使用函数形式能保证每次更新都基于最新的状态。

**setState**** 的回调函数**

setState 是异步的操作，我们并不能在执行完 setState 之后立马拿到最新的 state 的结果 , setState 支持传入一个回调函数，该回调函数会在状态更新并重新渲染后执行。

setState 的异步

  • 批量更新：React 为了提升性能，会对多次 setState 调用进行批量处理。在一个事件处理函数中多次调用 setState，React 会将这些更新合并成一次更新，从而减少不必要的重新渲染。
  • 异步更新：setState 是异步的，这意味着在调用 setState 之后，不能立即获取到更新后的状态。这是因为 React 可能会批量处理多个 setState 调用，为了保证性能和一致性，状态的更新是异步的。( 如果同步更新了 state，但是还没有执行 render 函数，那么 state 和 props 不能保持同步；state 和 props 不能保持一致性，会在开发中产生很多的问题 )

setState 一定是异步处理吗? (react18 前)

其实分成两种情况：

  • 在组件生命周期或 React 合成事件中，setState 是异步；
  • 在 setTimeout 或者原生 dom 事件中，setState 是同步

但是 在 react18 中更新了 Automatic Batching ,将所有事件都放入了批处理中 , 所有在 react18 后,都是异步处理的

  • 如果需要同步处理,可以通过 flushSync 实现

React Diff 算法是 React 协调（Reconciliation）过程的核心，它用于比较新旧虚拟 DOM 树，找出差异并将这些差异应用到真实 DOM 上，以此提高更新效率

完整比较两棵树的时间复杂度为 (O(n^3)) ，对于前端应用来说，这种复杂度是不可接受的。因此，React 基于以下三个假设来实现一个启发式的 (O(n)) 复杂度的 Diff 算法

  1. 不同类型的元素会产生不同的树：如果两个元素的类型不同，React 会直接销毁旧元素及其子元素，然后创建新元素及其子元素。
  2. 可以通过 **key** 来稳定标识子元素：当在数组中渲染多个子元素时，可以给每个子元素添加一个唯一的 key 属性，这样 React 就能识别出哪些元素被添加、删除或移动了。
    • **key**** 的选择**：key 应该是稳定、唯一且可预测的。通常可以使用数据的 ID 作为 key，避免使用数组的索引作为 key，因为当数组的顺序发生变化时，索引会导致不必要的重新渲染。
  3. 只进行同层比较：React 只会对同一层级的元素进行比较，不会跨层级比较元素

PureComponent

React.PureComponent 是 React 提供的一个基类，用于创建纯组件，它是 React.Component 的一个子类，在性能优化方面有独特的作用

它自动实现了 shouldComponentUpdate 生命周期方法，在这个方法中，React.PureComponent 会对组件的 props 和 state 进行浅比较。如果新的 props 和 state 与旧的相比没有变化（浅比较结果为相等），组件就不会重新渲染，以此避免不必要的渲染操作，提升性能。

浅比较是指只比较对象或数组的引用是否相同，而不深入比较对象或数组内部的属性值。具体规则如下

  • 对于基本数据类型（如 number、string、boolean 等），直接比较它们的值是否相等。
  • 对于引用数据类型（如 object、array 等），比较它们的引用是否相同，而不比较内部的属性值。
  • 浅比较的局限性：由于 React.PureComponent 只进行浅比较，当 props 或 state 是引用数据类型，且内部属性发生变化但引用未改变时，组件不会重新渲染[ 所以 state 要求必须为不可变数据 ]

函数组件替代方案：对于函数组件，可使用 React.memo 来实现类似的功能。React.memo 是一个高阶组件，它会对组件的 props 进行浅比较，若 props 没有变化，就会跳过组件的重新渲染

redux核心思想

React 中就要求我们无论是函数还是 class 声明一个组件，这个组件都必须像纯函数一样，保护它们的 props 不被修改 , redux 中，reducer 也被要求是一个纯函数

react-redux 是 React 官方推荐的用于管理应用状态的库，它是 Redux 在 React 中的绑定库。Redux就是一个帮助我们管理 State 的容器：Redux 是 JavaScript 的状态容器，提供了可预测的状态管理

Redux 的核心思想:

  • store
  • action
    • Redux 要求我们通过 action 来更新数据：
    • 所有数据的变化，必须通过派发（dispatch）action来更新； action 是一个普通的 JavaScript 对象，用来描述这次更新的 type 和 content；
    • 可以清晰的知道数据到底发生了什么样的变化，所有的数据变化都是可跟追、可预测的；
  • reducer
    • reducer 是一个纯函数；
    • reducer 做的事情就是将传入的 state 和 action 结合起来生成一个新的 state；

redux 三大原则

**单一数据源 **

  • 整个应用程序的 state 被存储在单一的 JavaScript 对象（即状态树）中，并且这个状态树是唯一的数据源
  • Redux 并没有强制让我们不能创建多个 Store，但是那样做并不利于数据的维护；
  • 单一的数据源可以让整个应用程序的 state 变得方便维护、追踪、修改；

**State 是只读的 **

  • 唯一修改 State 的方法一定是触发 action，不要试图在其他地方通过任何的方式来修改 State：
  • 这样就确保了 View 或网络请求都不能直接修改 state，它们只能通过 action 来描述自己想要如何修改 state；
  • 这样可以保证所有的修改都被集中化处理，并且按照严格的顺序来执行，所以不需要担心 race condition（竟态）的问题；

**使用纯函数来执行修改 **

  • 通过 reducer 将 旧 state 和 actions 联系在一起，并且返回一个新的 State：
  • 随着应用程序的复杂度增加，我们可以将 reducer 拆分成多个小的 reducers，分别操作不同 state tree 的一部分；
  • 但是所有的 reducer 都应该是纯函数，不能产生任何的副作用；

Hooks

认识Hooks

Hook 是 React 16.8 的新增特性，它可以让我们在不编写 class 的情况下使用 state 以及其他的 React 特性（比如生命周期）。

我们先来思考一下 class 组件相对于函数式组件有什么优势？比较常见的是下面的优势：

  • class 组件可以定义自己的state，用来保存组件内部状态；
    • 函数式组件不可以，因为函数每次调用都会产生新的临时变量；
  • class 组件有自己的生命周期，我们可以在对应的生命周期中完成自己的逻辑；
    • 比如在 componentDidMount 中发送网络请求，并且该生命周期函数只会执行一次；
    • 函数式组件 hooks 之前，如果在函数中发送网络请求，意味着每次重新渲染都会重新发送一次网络请求；
  • class 组件可以在状态改变时只会重新执行 render 函数以及我们希望重新调用的生命周期函数 componentDidUpdate 等；
    • 函数式组件在重新渲染时，整个函数都会被执行，似乎没有什么地方可以只让它们调用一次；

所以，在 Hook 出现之前，对于上面这些情况我们通常都会编写 class 组件。

类组件的缺陷

  • 复杂组件变得难以理解：
    • 我们在最初编写一个 class 组件时，往往逻辑比较简单，并不会非常复杂。但是随着业务的增多，我们的 class 组件会变得越来越复杂； 比如 componentDidMount 中，可能就会包含大量的逻辑代码：包括网络请求、一些事件的监听（还需要在 componentWillUnmount 中移除）而对于这样的 class 实际上非常难以拆分：因为它们的逻辑往往混在一起，强行拆分反而会造成过度设计，增加代码的复杂度；
  • 难以理解的 class：
    • 很多人发现学习 ES6 的 class 是学习 React 的一个障碍。 比如在 class 中，我们必须搞清楚 this 的指向到底是谁，所以需要花很多的精力去学习 this；
  • 组件复用状态很难：
    • 在前面为了一些状态的复用我们需要通过高阶组件； 像我们之前学习的 redux 中 connect 或者 react-router 中的 withRouter，这些高阶组件设计的目的就是为了状态的复用； 或者类似于 Provider、Consumer 来共享一些状态，但是多次使用 Consumer 时，我们的代码就会存在很多嵌套；

这些代码让我们不管是编写和设计上来说，都变得非常困难；

Hook 的出现，可以解决上面提到的这些问题；

它可以让我们在不编写 class 的情况下使用 state 以及其他的 React 特性； 但是我们可以由此延伸出非常多的用法，来让我们前面所提到的问题得到解决；

Hook 的使用场景：

  • Hook 的出现基本可以代替我们之前所有使用 class 组件的地方；
  • 但是如果是一个旧的项目，你并不需要直接将所有的代码重构为 Hooks，因为它完全向下兼容，你可以渐进式的来使用它；
  • Hook 只能在函数组件中使用，不能在类组件，或者函数组件之外的地方使用；

总结

  • 让函数式组件拥有状态、生命周期等特性的函数
  • 命名规则：所有 Hooks 以 **use** 开头（如 **useState**）
  • 使用规则：
    • 只能在函数组件或自定义 Hook 中调用
    • 必须位于组件顶层（不可嵌套在条件/循环中）

useState

useState 是 React 中最常用的 Hook 之一，它允许你在函数式组件中添加 state。

useState 函数接收一个初始值作为参数，并返回一个包含两个元素的数组：当前的 state 值和一个用于更新该 state 的函数。

初始值的形式

  • 静态初始值
  • 函数初始值

在 React 中，多次调用 setState（在类组件中）或 setCount（在函数式组件中使用 useState）时，React 会将这些更新合并。

如果新的 state 值依赖于之前的 state 值，建议使用函数式更新

useState 会帮助我们定义一个 state 变量，useState 是一种新方法，它与 class 里面的 this.state 提供的功能完全相同。

  • 一般来说，在函数退出后变量就会”消失”，而 state 中的变量会被 React 保留。
  • useState 接受唯一一个参数，在第一次组件被调用时使用来作为初始化值。（如果没有传递参数，那么初始化值为 undefined）。
  • useState 的返回值是一个数组，我们可以通过数组的解构，来完成赋值会非常方便

FAQ：为什么叫 useState 而不叫 createState?

  • “create” 可能不是很准确，因为 state 只在组件首次渲染的时候被创建。
  • 在下一次重新渲染时，useState 返回给我们当前的 state。
  • 如果每次都创建新的变量，它就不是 “state”了。
  • 这也是 Hook 的名字总是以 use 开头的一个原因。

useEffect

useEffect 是 React 中一个非常重要的 Hook，它用于处理函数组件中的副作用操作。副作用是指那些在组件渲染之外执行的操作，像数据获取、订阅、手动 DOM 操作等。

基本用法

useEffect 接收两个参数：一个回调函数和一个可选的依赖项数组。

  • 回调函数：每次组件渲染后都会执行该回调函数中的代码。
  • 依赖项数组：它是可选的。如果传入了依赖项数组，useEffect 会根据数组中的值来决定是否执行回调函数。
  • 执行时机** : **useEffect 的回调函数会在组件渲染到屏幕之后执行。这意味着它不会阻塞浏览器的渲染过程，保证了页面的流畅性。

清理函数

useEffect 的回调函数可以返回一个清理函数。清理函数会在组件卸载时执行，也会在每次 useEffect 重新执行之前执行，用于清理之前的副作用。

依赖项

  • 没有依赖项数组
    • 如果不传入依赖项数组，useEffect 的回调函数会在每次组件渲染后都执行。
  • 空依赖项数组
    • 当传入一个空的依赖项数组时，useEffect 的回调函数只会在组件挂载（首次渲染）和卸载时执行。
  • 有依赖项数组
    • 当传入一个包含值的依赖项数组时，useEffect 的回调函数会在组件挂载时执行，并且在依赖项数组中的任何一个值发生变化时再次执行。

useContext

useContext 是 React 提供的一个 Hook，其主要用途是在组件树中共享数据

  • 上下文的更新：当上下文的值发生改变时，所有使用该上下文的组件都会重新渲染。因此，要谨慎更新上下文的值，避免不必要的渲染。
  • 性能问题：如果上下文的值频繁变化，可能会导致性能问题。在这种情况下，可以考虑使用 useMemo 或 useCallback 来优化。

通过使用 useContext，你能够更便捷地在组件树中共享数据，减少 props 传递的复杂度，提升代码的可维护性。

useReducer

useReducer 是 React 中一个用于管理复杂状态逻辑的 Hook，它是 useState 的替代方案，特别适用于处理多个子值或者下一个状态依赖于之前状态的情况。

useReducer 基于 Redux 中 reducer 的概念，reducer 是一个纯函数，它接收当前状态和一个动作（action）作为参数，返回一个新的状态。在 React 里，useReducer 可以帮助你将状态更新逻辑集中管理。

  • useState：适用于简单的状态管理，状态更新逻辑相对简单，并且状态之间没有复杂的依赖关系。
  • useReducer：适用于复杂的状态管理，状态更新逻辑复杂，需要集中管理和处理多个状态更新。

useCallback和useMemo

useCallback 和 useMemo 都是 React 中的 Hooks，它们的主要作用是优化性能，避免不必要的计算和组件重新渲染

useCallback 用于缓存函数，它返回一个记忆化的回调函数。也就是说，在组件重新渲染时，如果依赖项没有发生变化，useCallback 返回的函数会保持不变，而不是重新创建一个新的函数。

正常情况下,我们的 jsx 组件中定义的函数会被重新定义一次,原来的函数会被销毁,因为 useState 每次数据变化,会生成一个新的 state 存储,并不是直接修改原 state,所以函数指向消失,函数被销毁,每次渲染会被重新定义一个新的函数

useCallback 接收两个参数：

  • 回调函数：需要记忆化的函数。
  • 依赖项数组：一个数组，包含了影响回调函数的变量。只有当依赖项数组中的变量发生变化时，useCallback 才会返回一个新的函数。

使用场景

  • 性能优化：当将回调函数作为 props 传递给子组件时，如果不使用 useCallback，每次父组件重新渲染时，传递的回调函数都会是一个新的函数，这可能会导致子组件不必要的重新渲染。使用 useCallback 可以避免这种情况。

useMemo

useMemo 是 React 提供的一个 Hook，主要用于性能优化，避免在每次渲染时都进行高开销的计算。

在 React 组件里，每次渲染时都会执行组件函数内部的代码。若某些计算操作较为复杂、耗时，而这些计算结果在多次渲染间并未改变，那么每次渲染都重新计算就会造成性能浪费。useMemo 能对计算结果进行缓存，只有当依赖项发生变化时，才会重新计算，从而避免不必要的重复计算，提升性能。

  • 避免复杂计算: 当组件中有一些计算开销较大的操作，如对大量数据进行排序、过滤等，可使用 useMemo 来缓存计算结果
  • **子组件优化: **若将一个复杂的对象或函数作为 props 传递给子组件，使用 useMemo 可以确保在依赖项不变时，传递给子组件的 props 保持不变，从而避免子组件不必要的重新渲染。

useRef

useRef 能创建一个可变的 ref 对象，此对象在组件的整个生命周期内保持不变。ref 对象有一个名为 current 的属性，你可以对其进行读取和赋值操作。和状态（state）不同，ref 对象的变化不会触发组件重新渲染。

  • useRef 接收一个可选的初始值作为参数，该初始值会被赋值给 ref 对象的 current 属性。
  • myRef.current 可用于获取或修改 ref 对象的值。

使用场景

  • 访问 DOM 元素: 借助 useRef 能够直接访问 DOM 元素，进而实现一些 DOM 操作，如聚焦输入框、获取元素尺寸等。
  • 保存可变值 :useRef 可用于保存一些在组件多次渲染间需要保持不变的可变值，避免闭包问题。
  • **避免闭包陷阱 **: 在使用回调函数时，可能会遇到闭包问题，导致回调函数引用的是旧的状态值。使用 useRef 可以解决这个问题。
